# 触发回归场景

下面这些场景构成插件式安装的公开触发契约。

自动强触发依赖仓库根目录里的插件文件，例如 `.codex-plugin/plugin.json`、`hooks.json` 和 `hooks/`。放在 `CODEX_HOME/skills` 下的 skill 式安装属于手动兜底路径，本身不会自动运行这些检查点。

自动强触发是固定检查点驱动的，不应该依赖 Codex 在任务快结束时才“想起”清理。如果自动 hook 不可用，手动兜底方式就是让 Codex 使用 `$codex-cleaning-temporary-processes`，或者直接运行脚本。

## 场景 1：一次性工具检查点已结束

- 发生了什么：某个已结束的步骤运行了一次性的 build、test、install、preview 或 runtime 命令，例如 `pnpm test`、`cargo test`、`pytest` 或 `dotnet build`。
- 插件式安装是否应该自动触发：是。
- 何时触发：在该检查点已经明确结束之后立刻触发，即使更大的任务还在继续。
- 预期模式：优先 `checkpoint-cleanup`；如果下一步仍可能复用该进程，则先 `inspect`。

## 场景 2：DevTools 或浏览器自动化检查点已结束

- 发生了什么：会话中使用了 `mcp__chrome_devtools__*`、`chrome-devtools-mcp`、Playwright 风格工具，或者使用带 `--remote-debugging-port` / `--headless` 的浏览器。
- 插件式安装是否应该自动触发：是。
- 何时触发：在 DevTools 或浏览器调试检查点结束之后触发，而不是只等到会话结束。
- 预期模式：对明显的 launcher、watchdog、helper shell 和远程调试残留，优先使用 `checkpoint-cleanup`；如果仍有复用可能，则先 `inspect`。第一次后续检查只有在当前线程显式确认“刚刚结束的检查点确实使用过这类显式自动化”，并同时提供非空工作区时，才会记录当前线程归属。

## 场景 3：helper 积压缓解检查点

- 发生了什么：重复的一批 shell 或工具命令已经结束，可能留下了临时 helper、wrapper 或 runtime 残留。
- 插件式安装是否应该自动触发：是。
- 何时触发：在这批命令结束之后、进程栈继续增长之前触发。
- 预期模式：如果归属仍有歧义，优先 `inspect`；如果“步骤已经结束”的证据足够强，再使用 `checkpoint-cleanup`。

## 场景 4：子代理检查点已结束

- 发生了什么：子代理已经完成工作，可能启动过 shell、runtime、测试、构建、浏览器 helper 或开发服务。
- 插件式安装是否应该自动触发：是，只要这些进程对当前检查点已经不再需要。
- 何时触发：在收到子代理结果、并且可以确认这些工具链已经完成当前步骤之后触发。
- 预期模式：如果仍可能复用，优先 `inspect`；如果这些残留已经明确是临时且已结束的，再使用 `checkpoint-cleanup`。

## 场景 5：会话结束检查点

- 发生了什么：会话正在结束，剩余的临时进程树已经不再需要。
- 插件式安装是否应该自动触发：是。
- 何时触发：在会话结束这个最终固定检查点触发。
- 预期模式：在前面的 checkpoint-safe 清理已经处理完阶段性残留之后，再用完整 `cleanup` 做最终清扫。

## 安全提醒

- 自动强触发必须继续保持保守。
- 当前线程拥有权只会收窄显式自动化的回收范围，不会放宽普通 runtime 的清理范围。
- 没有强任务证据时，不要清理普通交互 shell、普通用户浏览器、可复用开发服务或存在歧义的 runtime。
- 仅有工作区匹配仍然不足以清理显式自动化。
- 手动兜底必须和插件式安装保持同样的固定检查点与安全规则。
