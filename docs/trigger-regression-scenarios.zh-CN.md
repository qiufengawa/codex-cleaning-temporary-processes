# 触发回归场景

下面这些场景构成纯 skill 包的公开触发契约。

因为这个仓库是 skill，而不是插件，所以下面的检查点描述的是最佳努力的隐式调用目标，不是宿主保证的自动回调。

## 场景 1：一次性工具检查点已经结束

- 发生了什么：某个已结束步骤运行了一次性命令，例如 `npm test`、`vite build`、`vitest`、`cargo test`、`tauri build`、`pytest` 或 `dotnet build`。
- skill 是否应该考虑触发：是，最佳努力。
- 何时触发：一旦能明确确认这个检查点已经结束，就应该重新评估，即使更大的任务还在继续。
- 预期模式：优先使用 `checkpoint-cleanup`；如果下一步仍可能复用这些进程，就先 `inspect`。

## 场景 2：DevTools 或浏览器自动化检查点已经结束

- 发生了什么：当前会话使用过 `mcp__chrome_devtools__*`、`chrome-devtools-mcp`、Playwright 风格工具，或者使用带远程调试 / 无头标记的浏览器。
- skill 是否应该考虑触发：是，最佳努力。
- 何时触发：在显式自动化检查点结束后重新评估，而不是只等整个任务结束。
- 预期模式：对明显的 launcher、watchdog、helper shell 和远程调试残留，优先使用 `checkpoint-cleanup`；如果仍可能复用，就先 `inspect`。第一次后续检查只有在当前线程显式确认“刚结束的检查点确实使用过这类显式自动化”，并同时提供非空工作区时，才会记录当前线程归属。

## 场景 3：helper 积压缓解检查点

- 发生了什么：一批 shell 或工具命令已经结束，可能留下临时 helper、wrapper 或 runtime 残留。
- skill 是否应该考虑触发：是，最佳努力。
- 何时触发：在这批命令结束之后、进程继续堆积之前。
- 预期模式：如果归属仍然有歧义，就优先 `inspect`；如果“步骤已经结束”的证据足够强，再使用 `checkpoint-cleanup`。

## 场景 4：子代理检查点已经结束

- 发生了什么：子代理已经完成工作，可能留下 shell、runtime、测试、构建、浏览器 helper 或开发服务。
- skill 是否应该考虑触发：是，最佳努力。
- 何时触发：在收到子代理结果，并且可以确认它启动的工具链已经完成当前步骤之后。
- 预期模式：如果仍可能复用，就优先 `inspect`；如果这些残留已经明确是临时且已结束的，再使用 `checkpoint-cleanup`。

## 场景 5：最终清扫检查点

- 发生了什么：当前工作分支准备暂停，或者用户明确要求做一次最终清扫，而且剩余的临时进程树已经不再需要。
- skill 是否应该考虑触发：是，最佳努力。
- 何时触发：在能够明确确认“现在做最终清扫是安全的”这个时点。
- 预期模式：在前面的检查点清理已经处理完阶段性残留之后，再用完整 `cleanup` 做最终清扫。

## 安全提醒

- 最佳努力的隐式调用必须继续保持保守。
- 当前线程拥有权只会收窄显式自动化的回收范围，不会放宽普通 runtime 的清理范围。
- 没有强任务证据时，不要清理普通交互 shell、普通用户浏览器、可复用开发服务或有歧义的 runtime。
- 仅有工作区匹配仍然不足以清理显式自动化。
